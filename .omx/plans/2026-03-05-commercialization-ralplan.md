# FortuneLog 상용화 RALPLAN (Consensus, Deliberate)
- 작성일: 2026-03-05
- 기준 근거: `docs/functional-spec.ko.md`, `docs/tech-spec.ko.md`, `docs/fortune-service-ux-redesign-guide.md`, `.github/workflows/engine-test.yml`, 코드 스캔/테스트 실행 결과

## 1) 현재 진척도 진단 (상용화 준비도)

### 기능(Feature)
- 구현됨(중~상): 온보딩, 인증(이메일/소셜), 출생정보 CRUD, 사주 계산, 오늘 운세, AI 해석, 기본 마이페이지
- 부분 구현(중): 상세 리포트 UX는 존재하나 일부 정적/테스트 성격(실데이터 연동 일관성 부족)
- 미완/취약(하): 결제/구독 실연동, 주문/구독 상태 실시간 동기화, 탈퇴, 정책 링크 실서비스 연결
- 준비도: **약 65%**

### 품질(Quality)
- 강점: 엔진 도메인/인프라 테스트 존재, 현재 `./gradlew test` 통과
- 취약: 모바일 테스트가 단일 위젯 테스트 중심(`flutter test` 1개), E2E/회귀/릴리즈 품질게이트 부족
- 준비도: **약 40%**

### 운영(Ops/Release)
- 강점: Supabase 마이그레이션/스크립트 존재
- 취약: CI가 engine만 포함, 모바일 빌드/배포 자동화 부재, 관측(Sentry/Amplitude/알람) 실연동 근거 부족
- 준비도: **약 30%**

### 컴플라이언스/리스크(Compliance)
- 강점: RLS/권한 기반 기본 구조 있음
- 취약: 약관/개인정보/환불 정책의 실제 연결 및 탈퇴/파기 플로우 부재
- 준비도: **약 35%**

## 역할 코드명 매핑 (Avengers)
- Product Owner: 캡틴 아메리카
- Tech Lead/Architect: 아이언맨
- Mobile Lead: 스파이더맨
- Backend Lead: 비전
- Infra/DevOps: 워머신
- QA Lead: 블랙 위도우
- Data/Analytics: 브루스 배너
- Security/Compliance/Legal: 블랙 팬서
- Release Manager: 호크아이

## 2) 갭 분석 (MVP 및 상용화 기준 대비)
- MVP 결제/구독: 스펙상 필수이나 현재는 DB 스키마 중심, 앱/웹훅/상태반영의 제품 플로우 미완성
- 리포트 완성도: 생성 API는 있으나 사용자 노출 화면 일부가 정적 데이터 중심
- 리텐션 핵심: 푸시(오늘 운세) 설정/발송/클릭 트래킹 체인 미완성
- 상용 운영: 장애 탐지/성능 지표/비용 지표/릴리즈 게이트 부족
- 출시 필수 정책: 약관/개인정보/환불 링크 및 탈퇴 처리 미완성

### 코드 근거 스냅샷 (Evidence)
- 결제/구독 UI가 정적 상태 중심: `apps/mobile/lib/features/mypage/my_page.dart`
- AI 탭 테스트 문구 존재(정식 결제 전): `apps/mobile/lib/features/dashboard/dashboard_page.dart`
- 엔진 API 범위(차트/리포트/AI해석/오늘운세/헬스): `services/engine-api/src/main/java/com/fortunelog/engine/api/EngineController.java`
- 모바일 자동화 테스트 최소 수준(단일 위젯 테스트): `apps/mobile/test/widget_test.dart`
- CI는 엔진 테스트만 확인: `.github/workflows/engine-test.yml`
- MVP 요구 범위(결제/구독/푸시/탈퇴 포함): `docs/functional-spec.ko.md`

### 코드 품질 리뷰 요약 (2026-03-05, worker-3)
- **Critical**: CI가 엔진 테스트 중심으로 구성되어 모바일 회귀/빌드 실패를 PR 단계에서 조기 차단하지 못함
- **Major**: 결제/구독 화면의 정적 상태 의존으로 실제 주문/권한 동기화 실패가 UX에 즉시 노출되지 않을 수 있음
- **Major**: 모바일 자동화 테스트 범위가 단일 위젯 테스트 수준이라 상용 릴리즈 회귀 방어력이 부족함
- **Major**: 결제 웹훅 지연/중복/순서역전 시나리오에 대한 상태 검증 자동화가 문서화 대비 구현 근거가 약함
- **Minor**: 일부 수용 기준은 측정 지표가 있으나 실행 명령/담당 역할 연결(누가, 어떤 커맨드로 검증하는지)이 추가되면 운영 전환성이 개선됨

### 문서 보강 액션 (품질 관점)
- Phase 1 완료 산출물에 **모바일 CI 게이트 추가 상태**를 명시적으로 기록
- Phase 2 수용 기준에 **정책 링크 모니터링 방법(체크 주기/알람 기준)** 문구를 운영 문서와 연결
- Phase 3 보안 점검 항목에 **RLS 회귀 테스트 명령/담당자**를 릴리즈 체크리스트와 교차참조

## 3) 추가 필요 기능 백로그 (P0/P1/P2)

### P0 (출시 전 필수)
- 결제 실연동 + 서버 검증 + 주문/구독 상태 동기화
- 유료 권한 반영(앱 UI + DB + 서버 접근제어 일관 적용)
- 탈퇴/데이터 파기(or 비식별화) 플로우
- 정책 링크(약관/개인정보/환불) 실제 URL 연결
- 모바일 CI(테스트 + 빌드) 및 릴리즈 체크게이트
- 관측 최소세트: 크래시, 핵심 API 지연/에러율, 결제 실패율 알람

### P1 (출시 직후 1~2 스프린트)
- 푸시 알림 체인 완성(설정, 스케줄, 발송, 클릭 이벤트)
- 상세 리포트 3탭 완전 실데이터화 + 재생성 UX 안정화
- 결제 실패/유예(grace) 복구 플로우(재결제, 결제수단 변경, 문의)
- 이벤트 퍼널 대시보드(가입→계산→리포트→결제)

### P2 (성장/최적화)
- AI 해석 캐시/쿼터/비용 가드레일 고도화
- A/B 테스트(페이월 카피, 온보딩, CTA)
- 성능 튜닝(앱 초기 로드, 차트/리포트 조회 지연)
- 운영도구(관리자 조회/재처리/CS 대응) 강화

## 4) 우선순위 실행계획 (Phase/Milestone)

### Phase 0. 상용화 기준선 확정 (1주)
- 오너: 캡틴 아메리카(Product) + 아이언맨(Tech Lead) + 비전(Backend)
- 목표: 결제 제공자, 출시 플랫폼, KPI/오류예산, D-데이 확정
- 작업:
  - 결제 아키텍처 확정(RevenueCat vs 직접 웹훅)
  - 결제 상태 단일 진실원천(Source of Truth) 확정:
    - 앱 표시 기준 = `subscriptions/orders` 스냅샷
    - 상태 전이 기준 = provider webhook verified event
  - 멱등/재처리 규칙 확정:
    - idempotency key = `provider + provider_order_id + event_id`
    - 재처리 큐(DLQ 포함)와 수동 재처리 런북 정의
  - 릴리즈 Go/No-Go 체크리스트 확정
  - 이벤트/지표 정의서 확정(가입→계산→리포트→결제)
- 수용 기준:
  - ADR 승인 1건
  - 상용 KPI 5개(전환/리텐션/실패율/응답속도/크래시율) 문서화
  - 결제 상태전이 다이어그램(정상/중복/역전/지연 이벤트) 승인
- 의존성: 없음
- 리스크: 의사결정 지연
- 완료 산출물:
  - 결제 ADR + 상태전이 시퀀스 다이어그램
  - 상용화 Go/No-Go 체크리스트 v1

### Phase 1. 매출 경로 + 관측 최소세트 완성 (2~3주)
- 오너: 스파이더맨(Mobile) + 비전(Backend) + 브루스 배너(Data) + 워머신(Infra)
- 목표: 결제/구독 실연동 + 권한 반영 + 관측 최소세트 확보
- 작업:
  - 결제 시작/검증/주문상태 업데이트 플로우 구현
  - 구독 상태(active/grace/expired/canceled) 동기화 및 UI 반영
  - 유료 리포트 접근제어(서버 기준) 적용
  - 관측 최소세트 적용:
    - Mobile/Backend Sentry
    - 결제 실패율/권한반영 지연/핵심 API P95 대시보드
    - `requestId` 기반 로그 상관추적
- 수용 기준:
  - 샌드박스 결제 3회 연속 성공/취소/실패 케이스 통과
  - 결제 후 5초 내 권한 반영
  - 주문/구독 화면이 DB 실데이터 기반으로 동작
  - 결제 실패율 알람(5분 이동창 > 3%) 발화 검증 1회 이상
- 의존성: Phase 0 결제 결정
- 리스크: 스토어 검증 지연, 웹훅 멱등성 누락
- 완료 산출물:
  - 결제 웹훅 runbook(중복/지연/실패 재처리 포함)
  - 상용 대시보드 URL/알람 정책 문서

### Phase 2. 핵심 경험 + 신뢰 경로 완성 (2주)
- 오너: 스파이더맨(Mobile) + 비전(Backend)
- 목표: 리포트/운세 경험 일관성 + 정책/탈퇴 경로 구현
- 작업:
  - 상세 리포트 화면 실데이터 연동 정합성 완료
  - 오늘 운세 생성/조회 실패 복구 UX 정리
  - 푸시 토대(발송 스케줄 + 앱 설정) 1차 구현
  - 정책 문서 URL(약관/개인정보/환불) 앱 실제 연결
  - 탈퇴/파기 플로우 API 및 UX 1차 구현(soft-delete + 비식별화 큐)
- 수용 기준:
  - 상세 리포트 3탭 모두 실데이터 표시
  - 오늘 운세 생성 실패율 < 2% (주간)
  - 푸시 설정/해제 및 발송 로그 확인 가능
  - 정책 링크 100% 유효 + 모니터링 가능
  - 탈퇴 요청 후 접근 차단 ≤ 1분, 파기 작업 큐 적재 확인
- 의존성: Phase 1 권한/상태 완료
- 리스크: 시차/날짜 기준 불일치(KST vs 로컬)
- 완료 산출물:
  - 정책 링크 검증 리포트
  - 탈퇴/파기 동작 시퀀스 및 운영 가이드

### Phase 3. 운영/품질 게이트 (2주)
- 오너: 블랙 위도우(QA) + 워머신(Infra) + 스파이더맨(Mobile) + 비전(Backend)
- 목표: 출시 전 장애/품질/보안 최소 기준 충족
- 작업:
  - 모바일 CI(테스트/빌드), 백엔드 CI 확장
  - Sentry/로그 상관관계(requestId) 대시보드 고도화
  - 보안 점검(JWT 설정, 키 관리, RLS 회귀)
- 수용 기준:
  - CI: 모바일/엔진 모두 PR 필수 통과
  - 크래시프리 세션 > 99.5%(사내 베타)
  - P95 API 지연 < 2s (핵심 엔드포인트)
- 의존성: Phase 1~2 기능 고정
- 리스크: 관측지표 누락으로 원인파악 지연
- 완료 산출물:
  - 모바일/백엔드 CI 게이트 정의서
  - 보안 점검 체크리스트 결과표

### Phase 4. 출시 준비/컴플라이언스 승인 (1주)
- 오너: 캡틴 아메리카(Product) + 블랙 팬서(Legal/Compliance) + 호크아이(Release)
- 목표: 스토어 제출 가능한 운영/법무 패키지 완성
- 작업:
  - 정책/탈퇴 플로우 최종 QA 및 법무 승인
  - 앱스토어/플레이스토어 메타데이터, 스크린샷, 심사노트 준비
- 수용 기준:
  - 정책/탈퇴 회귀 테스트 100% 통과
  - 릴리즈 체크리스트 전 항목 완료
- 의존성: 전 단계 안정화
- 리스크: 법무 승인/스토어 심사 반려
- 완료 산출물:
  - 스토어 제출 패키지(메타데이터/심사노트)
  - 릴리즈 승인 기록(Go/No-Go 서명)

### Cross-Phase. Guarded Beta Gate (병행, 2주)
- 오너: 캡틴 아메리카(Product) + 블랙 위도우(QA) + 워머신(Infra)
- 목표: 제한 사용자 기반 실사용 학습과 안정성 검증 동시 달성
- 규칙:
  - 대상 사용자: 내부/초대 베타 최대 300명
  - 롤아웃 단계: 5% → 20% → 50% → 100%
  - 롤백 트리거:
    - 결제 실패율 > 2% (30분)
    - 권한 반영 지연 P95 > 15초 (30분)
    - crash-free < 99.3% (24시간)
  - 승급 조건:
    - 각 단계 최소 24시간 이상 기준치 유지
    - Sev1/Sev2 신규 이슈 0건
- 완료 산출물:
  - 단계별 승급/롤백 로그
  - 베타 학습 리포트(전환/이탈/오류 원인)

## 5) RALPLAN-DR 요약

### Principles (원칙)
- 매출 경로(결제/권한) 우선 완성
- 보안/데이터 정합성은 기능보다 선행 검증
- 운영 가시성 없는 기능은 출시 불가
- 사용자 신뢰(정책/탈퇴/오류복구)를 제품 기능으로 취급
- 단계별 릴리즈(가드레일 기반)로 리스크 격리

### Decision Drivers (Top 3)
- 출시까지 리드타임 단축
- 결제/구독 오류 리스크 최소화
- 장애 발생 시 복구/원인추적 가능성 확보

### Viable Options
- 옵션 A: 기능 선출시 후 운영/결제 보강
  - 장점: 빠른 출시
  - 단점: 결제/권한 오류 시 신뢰와 매출 동시 훼손
- 옵션 B: 결제-권한-관측 선완성 후 단계 출시
  - 장점: 상용 리스크를 초기에 제거, 운영 가능성 높음
  - 단점: 초기 1~2주 체감 속도 저하
- 옵션 C: 전면 재설계 후 출시
  - 장점: 장기 구조 일관성
  - 단점: 시간/범위 과다, 현재 자산 활용도 낮음

### Chosen Option
- **옵션 B 채택**
- 단, 실행 방식은 **Guarded Beta 병행형 옵션 B**로 확장

### Invalidation Rationale
- 옵션 A 기각: 상용화 초기 가장 치명적인 실패는 결제/권한 불일치이며, 현 상태에서 선출시하면 리스크가 과대
- 옵션 C 기각: 현재 코드/스키마 자산이 충분하여 전면 재설계는 ROI가 낮고 일정 리스크가 큼

## 6) Deliberate Mode 보강

### Pre-mortem (3 시나리오)
- 시나리오 1: 결제 성공 후 유료 권한 미반영
  - 원인: 웹훅 지연/중복 처리, 상태 동기화 race
  - 대응: 멱등 키, 재처리 큐, 권한 재동기화 버튼
- 시나리오 2: 인증/JWT 환경오류로 실사용자 401 급증
  - 원인: 환경별 키/검증 경로 불일치
  - 대응: 배포 전 auth smoke test, 환경 diff 체크 자동화
- 시나리오 3: AI 비용/지연 급증으로 UX 악화
  - 원인: 요청량 증가, 타임아웃/재시도 과다
  - 대응: 캐시, fallback 전략, 비용 알람/쿼터 제한

### Expanded Test Plan
- Unit
  - 결제 상태 매핑, 권한 판정, 날짜/타임존 로직, 에러 매퍼
- Integration
  - 앱↔엔진↔Supabase↔웹훅 end-to-end 상태 반영
- E2E
  - 가입→출생정보→차트→리포트→결제→유료열람→재방문
- Observability
  - requestId 추적, 실패율/지연 대시보드, 알람 룰 검증
- Security
  - JWT 검증 모드, 키 노출 점검, RLS 회귀 테스트, 민감정보 로깅 점검
- Release
  - 모바일 빌드/서명/스토어 제출 파이프라인 dry-run

### Verification Steps (구체 검증 절차)
- 결제 일관성:
  - 샌드박스에서 정상/중복/순서역전 webhook 이벤트를 재생하고 `orders/subscriptions` 최종 상태 일치 검증
- 권한 반영:
  - 결제 완료 시점부터 유료 화면 접근 가능 시점까지 100회 샘플링, P95 < 5초 확인
- 운영 가시성:
  - 의도적 401/500/타임아웃 주입 후 Sentry 이벤트, 알람, requestId 추적 연계 확인
- 컴플라이언스:
  - 탈퇴 요청 후 로그인 차단, 데이터 조회 불가, 파기 큐 처리 로그를 QA 시나리오로 검증
- 릴리즈:
  - iOS/Android 사전 배포 채널에서 빌드/서명/배포 dry-run 1회 이상 통과

## 7) ADR
- Decision
  - 상용화 전략으로 "결제-권한-관측 최소필수 선완성 + Guarded Beta 단계 출시"를 채택
- Drivers
  - 매출 보호, 장애 대응력, 일정 예측 가능성
- Alternatives considered
  - A 기능 선출시, C 전면 재설계
- Why chosen
  - 현재 코드 자산을 유지하면서 상용 리스크를 가장 크게 줄임
- Consequences
  - 초기 1~2주 기능 확장 속도보다 안정화 투자 비중 증가
  - 대신 제한 베타에서 학습 속도를 유지하며 리스크를 관리 가능
- Follow-ups
  - 결제 제공자 확정, 모바일 CI 구축, 정책/탈퇴 구현, 베타 런치 지표 관찰
  - 결제 멱등/재처리 런북과 대시보드 URL을 릴리즈 문서에 연결

## 8) 합의 검토 로그 (Planner → Architect → Critic)
- Planner 초안: 기능/운영/컴플라이언스 분리 및 5단계 로드맵 제안
- Architect 1차 판정: **NEEDS_REVISION**
  - 이유: 옵션 B 정의 대비 관측 시점 지연, 결제 멱등/재처리 제약 부족, 신뢰 경로(정책/탈퇴) 후순위
- Planner 수정 반영:
  - 관측 최소세트 Phase 1 선행
  - 결제 Source-of-Truth/멱등/DLQ 규칙 추가
  - 정책/탈퇴 경로를 Phase 2로 당김
  - Guarded Beta 수치 게이트(롤아웃/롤백) 추가
- Critic 최종 판정: **APPROVE (residual risk: 스토어 심사 일정 변동, 결제 provider 운영정책 변경)**

## 9) 문서 품질 DoD (Definition of Done) 체크
- [ ] 각 Phase 수용 기준마다 실행 커맨드/대시보드/로그 소스가 1개 이상 연결되어 있는가
- [ ] 결제/권한/관측 지표가 동일한 용어와 임계치로 문서 전반에서 일관되는가
- [ ] 정책/탈퇴/환불 경로가 앱 UI, API, 운영 런북에 모두 연결되어 있는가
- [ ] Guarded Beta 승급/롤백 기준이 알람 정책과 동일하게 구성되어 있는가
- [ ] 릴리즈 Go/No-Go 항목이 법무/보안/QA 승인 주체까지 명시되어 있는가
